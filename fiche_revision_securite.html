Lire un log auth.log — détecter une attaque SSH
# Extrait typique de /var/log/auth.log Jun 3 18:45:23 sshd[2890]: Failed password for invalid user admin from 203.0.113.5 port 48123 Jun 3 18:45:26 sshd[2891]: Failed password for root from 203.0.113.5 port 48125 Jun 3 18:45:28 sshd[2892]: Failed password for user1 from 203.0.113.5 port 48126
⚠️ Brute Force SSH détecté : même IP (203.0.113.5), tentatives répétées en quelques secondes, utilisateurs différents (admin, root, user1).
Réaction immédiate
Bloquer l'IP via iptables :
iptables -A INPUT -s 203.0.113.5 -j DROP
Outil automatique
Fail2ban — surveille les logs et bannit automatiquement les IP après N échecs configurables
Durcissement SSH — /etc/ssh/sshd_config
PermitRootLogin no PasswordAuthentication no PubkeyAuthentication yes Port 2222 MaxAuthTries 3
🔑
Clé publique : seule méthode vraiment sûre — la clé privée ne quitte jamais le client
🚫
PermitRootLogin no : même avec le bon mot de passe, root ne peut pas se connecter
🔢
Changer le port : réduit le bruit des scans automatiques (sécurité par l'obscurité, pas suffisant seul)
iptables — règles essentielles
ObjectifRègle iptables
Bloquer une IPiptables -A INPUT -s 203.0.113.5 -j DROP
Autoriser SSH entrantiptables -A INPUT -p tcp --dport 22 -j ACCEPT
Autoriser connexions établiesiptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
Bloquer tout le reste (INPUT)iptables -A INPUT -j DROP
SNAT (masquerade)iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE
DNAT (port forwarding)iptables -t nat -A PREROUTING -p tcp --dport 80 -j DNAT --to 10.0.0.10:80
💡 Ordre des chaînes iptables : PREROUTING → FORWARD/INPUT → POSTROUTING. Le NAT entrant (DNAT) va en PREROUTING, le NAT sortant (SNAT/MASQUERADE) va en POSTROUTING.

🟣 Chiffrement & Cryptographie
Symétrique vs asymétrique, certificats TLS, GPG, et les pièges classiques à l'exam.
Symétrique vs Asymétrique
Chiffrement symétriqueChiffrement asymétrique
Clés1 clé partagée (même pour chiffrer et déchiffrer)Paire : clé publique + clé privée
ExempleAES-256, DES, 3DES, ChaCha20RSA, ECC, GPG, TLS handshake
Vitesse⚡ Très rapide — adapté aux gros volumes🐢 Lent — réservé aux échanges de clés
ProblèmeComment partager la clé secrètement ?Aucun — la clé publique peut être distribuée librement
UsageChiffrement des données en transit ou au reposÉchange de clés, signature, authentification
💡 En pratique (TLS) : on utilise l'asymétrique pour échanger une clé de session, puis le symétrique pour chiffrer les données. Le meilleur des deux mondes.
GPG — chiffrement de fichiers
# Chiffrer un fichier pour Bob gpg --encrypt --recipient bob@example.com fichier.pdf # Bob déchiffre avec sa clé privée gpg --decrypt fichier.pdf.gpg
🔓
Chiffrement avec la clé publique de Bob → seul Bob peut déchiffrer (avec sa clé privée)
📦
Même si le fichier est intercepté en transit, il est illisible sans la clé privée
✉️
Indépendant du protocole de transport (SMTP, FTP, HTTP…)
TLS/SSL — comment ça fonctionne
1. Handshake
Client et serveur négocient la version TLS, les algorithmes et échangent les certificats
2. Certificat
Le serveur prouve son identité avec un certificat signé par une Autorité de Certification (AC)
3. Session chiffrée
Une clé de session symétrique (AES) est établie — toutes les données suivantes sont chiffrées
💡 Rôle du certificat TLS : authentifier le serveur (éviter le MITM) + permettre l'échange sécurisé de la clé de session. Sans certificat valide → ton navigateur affiche une alerte.
Pièges classiques à l'exam
⚠️
Déchiffrement ≠ Décryptage — déchiffrer = utiliser la clé légitime. Décrypter = retrouver le message sans la clé (attaque force brute).
⚠️
SMTP + TLS ≠ chiffrement de bout en bout — le TLS chiffre le transport mais le message peut être stocké en clair sur le serveur destinataire.
⚠️
Clé publique ≠ secret — elle peut être distribuée librement. C'est la clé privée qui doit rester secrète à tout prix.
⚠️
Ordinateur quantique — menace les algorithmes asymétriques (RSA, ECC) basés sur la factorisation. AES-256 reste résistant avec des clés suffisamment longues.
✅
GPG vs SMTP chiffré — GPG est préférable car le chiffrement est fait avant l'envoi, indépendamment du protocole.

🟡 DMZ, pfSense & Messagerie
Architecture DMZ, règles pfSense, serveur de messagerie sécurisé et RGPD.
Architecture DMZ — principe et règles
Internet ← WAN → [pfSense] ← DMZ → [Serveur Web] ↕ LAN (réseau interne)
Principe de cloisonnement : si le serveur web est compromis, l'attaquant ne peut pas atteindre le LAN car la DMZ est un réseau séparé avec des règles de filtrage strictes.
✅
Internet → DMZ port 443 : autorisé (accès HTTPS au serveur web)
✅
LAN → DMZ : autorisé avec contrôle (admins)
🚫
DMZ → LAN : toujours bloqué — le serveur web ne doit jamais initier de connexion vers le LAN
🚫
DMZ → Internet : bloqué par défaut sauf si nécessaire (mises à jour…)

Serveur de messagerie sécurisé — Postfix + Dovecot
Postfix (SMTP) — envoi
Port 25 : échanges inter-serveurs (MTA↔MTA)
Port 587 : soumission client authentifié (SASL + TLS)
Port 465 : SMTPS (déprécié, éviter)
Dovecot (IMAP) — réception
Port 143 : IMAP non chiffré (à éviter)
Port 993 : IMAPS (IMAP over TLS) ✅
🚫
Open relay — serveur SMTP qui accepte d'envoyer des mails pour n'importe qui sans authentification → spam massif, blacklist
✅
SASL obligatoire — impose une authentification avant tout envoi de mail
✅
TLS obligatoire — chiffrer le transport entre client et serveur
📋
RGPD — les logs de messagerie contiennent des données personnelles → durée de rétention limitée, accès restreint

🟢 Proxy & VPN
Rôle du proxy, types de VPN, tunnels site-à-site et détection des flux anormaux.
Proxy HTTP — rôles et effets
TypeComportement
Proxy transparentLe client ne le sait pas. Le trafic est redirigé automatiquement (port 80/443 → proxy). L'IP source vue par le serveur = IP du proxy.
Proxy expliciteLe client est configuré manuellement. Envoie ses requêtes au proxy qui les relaie.
Proxy inverse (reverse)Côté serveur. Reçoit les requêtes d'Internet et les redirige vers les serveurs internes. Rôle de load balancer + cache.
✅
Cache — stocke les réponses HTTP pour accélérer les accès répétés et réduire la bande passante
✅
Filtrage URL — bloque les sites non autorisés (listes noires, catégories)
✅
Logs — trace toutes les requêtes HTTP des utilisateurs (RGPD !)
⚠️
Contournement HTTPS — un tunnel HTTPS peut contourner le proxy si le filtrage SSL n'est pas configuré
VPN — types et usages
TypeUsageProtocoleCaractéristiques
VPN site-à-siteRelier deux réseaux distants (sièges, agences)IPsec, OpenVPNTunnel permanent, transparent pour les utilisateurs
VPN client-réseauTélétravail — un utilisateur accède au LAN distantOpenVPN, WireGuard, SSL VPNLogiciel sur le poste client
VPN split tunnelingSeul le trafic interne passe par le VPN, Internet passe directConfigurable⚠️ Risque : le trafic Internet n'est pas filtré par l'entreprise
💡 IPsec : fonctionne en couche 3 — chiffre les paquets IP directement. Deux modes : Transport (chiffre uniquement le payload) et Tunnel (encapsule le paquet IP entier dans un nouveau paquet).
VPN site-à-site — ce qu'il faut configurer
Côté Siège A (10.0.1.0/24)
IP publique : 203.0.113.1
Réseau local : 10.0.1.0/24
Route → 10.0.2.0/24 via tunnel
Côté Agence B (10.0.2.0/24)
IP publique : 198.51.100.1
Réseau local : 10.0.2.0/24
Route → 10.0.1.0/24 via tunnel
🔑
Authentification mutuelle — certificats ou clés pré-partagées (PSK) des deux côtés
🔒
Algorithmes — Phase 1 : IKE (échange de clés). Phase 2 : ESP (chiffrement AES) + authentification
📊
Supervision — surveiller l'état du tunnel (up/down), les logs IKE, la latence
🔄
Haute disponibilité — tunnel de secours sur une autre interface WAN (failover)

Architecture complète typique — à savoir schématiser
Internet | [Pare-feu WAN] ← pfSense / iptables | DMZ ──── Serveur Web (443) | Serveur Mail (25/587/993) | Proxy sortant | [Pare-feu interne] | LAN ──── Postes utilisateurs Serveurs internes Annuaire LDAP/AD | [VPN concentrateur] ←── Télétravail / Site distant
💡 À retenir pour l'exam : le proxy est dans la DMZ ou sur le pare-feu. Le VPN concentrateur est côté LAN ou sur le pare-feu. Les flux DMZ→LAN sont toujours bloqués par défaut.
